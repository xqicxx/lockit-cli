# Lockit Code Style

这是 `lockit` 项目的代码规范，适用于 Rust CLI、后续桌面端核心、同步层和协议层。

## 1. 设计原则

- 高内聚、低耦合。
- 一个模块只负责一个明确职责。
- 业务规则、存储、加密、CLI 输出要分层。
- 默认安全优先，所有 secret 都按高敏感数据处理。
- 代码要为未来的 CLI、桌面端、Android 共享协议留出余地。

## 2. 目录与职责

- `src/domain/`：凭据模型、类型、校验规则、策略。
- `src/application/`：用例编排，如 `add`、`get`、`use`、`sync`。
- `src/security/`：KDF、加密、解密、脱敏、密钥轮换。
- `src/vault/`：本地 vault 读写、版本迁移、manifest。
- `src/sync/`：WebDAV、Google Drive、冲突检测与同步协议。
- `src/cli/`：参数解析、子命令、帮助文本、输出格式。
- `src/integrations/`：env、dotenv、cookie、ssh 等外部导出能力。

## 3. Rust 编码规范

- 优先使用清晰的小函数，不要把多个业务步骤写进一个超长函数。
- 类型要表达意图，避免大量 `String`、`HashMap<String, String>` 直接穿透核心层。
- 错误使用明确的枚举或 `thiserror`，不要把所有错误都吞成 `anyhow!`。
- 能用 `Result` 表达失败的地方不要用 `panic!`。
- 需要共享的逻辑放进可测试的纯函数。
- 命名要能直接说明用途，避免 `data2`、`tmp`、`handle` 这类弱语义名字。

## 4. 安全规范

- 默认不输出完整 secret。
- 不把 secret 写进日志、错误信息、调试输出或临时文件。
- 加密前后的数据边界要清晰，解密后的明文生命周期尽量短。
- 云同步只传输加密内容，不传输明文凭据。
- 远程同步默认要求 HTTPS。

## 5. 模块边界规范

- CLI 只负责接收参数和展示结果，不负责业务细节。
- application 层只编排流程，不直接依赖具体文件格式。
- vault 层只处理本地存储，不知道 CLI 细节。
- sync 层只处理同步协议，不知道用户界面。
- domain 层不依赖 IO、网络和命令行参数。

## 6. 测试规范

- 核心逻辑必须能单测。
- 加解密必须有 round-trip 测试。
- 导入导出必须有边界测试。
- 同步 manifest 必须有序列化与反序列化测试。
- CLI 命令必须至少覆盖主流程和错误输入。

## 7. 提交前检查

- [ ] 这个改动是否只改了应该改的模块？
- [ ] 是否引入了 secret 明文泄露路径？
- [ ] 是否有更小、更清晰的函数拆分方式？
- [ ] 是否补了测试或至少说明了为什么无法补？
- [ ] 是否会让后续迁移到加密 vault 更难？

## 8. 代码风格约定

- 注释只解释“为什么这样做”，不重复“代码做了什么”。
- 公开接口要稳定、语义清晰。
- 格式化和 lint 结果应该保持一致。
- 任何新命令都要先考虑可测试性和安全输出。
