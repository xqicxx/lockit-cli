# Lockit Code Acceptance

## 1. 设计原则

优秀代码不只追求低耦合和可复用，还必须同时满足下面这些要求：

- 高内聚：每个模块只负责一类清晰问题，改动集中，不把无关逻辑揉在一起
- 可读性：命名达意、层次明确、控制流清晰，别人读到代码时能迅速理解意图
- 可维护性：新增功能时优先扩展而非推翻旧结构，避免“牵一发而动全身”
- 可测试性：核心逻辑可以脱离 CLI、文件系统、网络环境单独测试
- 健壮性：异常输入、空值、损坏数据、重复操作、并发冲突都能被优雅处理
- 性能合理：避免无意义的全量扫描、重复解密、重复序列化与网络往返
- 安全性：默认最小暴露面，敏感信息不明文泄露到日志、缓存、历史记录或错误输出
- 规范一致：错误处理、命名风格、目录结构、日志格式保持统一

## 2. 模块级验收标准

### 领域层
- 类型定义稳定，不依赖 CLI 或具体存储实现
- 关键字段具有明确约束与校验规则
- 不允许把多种 credential 类型粗暴塞进难以维护的 if/else 分支

### 应用层
- 一个 use case 只处理一个明确业务动作
- 业务编排与 IO 细节分离
- 所有错误都能返回可定位、可理解的上下文

### 基础设施层
- 文件系统、数据库、网络、云盘实现都有抽象接口
- 敏感操作具备超时、重试和错误分类
- 存储迁移具备版本识别和回滚思路

### CLI 接口层
- 默认输出对 secret 脱敏
- 参数错误时给出明确可执行的提示
- 命令帮助信息完整，示例清晰

## 3. 测试验收标准

### 单元测试
- `Credential` 校验规则有覆盖
- 加解密 round-trip 有覆盖
- manifest 序列化与反序列化有覆盖
- env / dotenv / cookie / ssh 导出器有覆盖

### 集成测试
- `init -> add -> get -> list -> delete` 主流程可跑通
- 错误主密码、损坏 vault、重复导入、缺失字段都有验证
- `run -- <command>` 能正确注入环境变量
- WebDAV 同步基础流程有自动化验证

### 回归测试
- schema 升级前后的兼容路径可验证
- Android / CLI 针对同一测试样本能读写一致数据

## 4. 安全验收标准

- 禁止默认明文落盘 secret
- 禁止在普通日志里打印完整 secret
- 禁止把 secret 放进 panic/error message 的上下文
- 解密后的敏感值生命周期要尽量短
- 同步前后都只传输加密内容
- WebDAV 等远端配置必须强制 HTTPS 或给出明确风险开关

## 5. 性能验收标准

- 凭据读取不应在每个命令路径上做不必要的重复解析
- 列表查询和按名称查询应具备清晰的复杂度边界
- 同步时只上传必要对象，避免重复大文件写入

## 6. Review Checklist

提交代码前至少确认以下问题：

- [ ] 这段代码的职责是否单一且清晰？
- [ ] 是否有更简单、边界更清楚的拆分方式？
- [ ] 是否可在不依赖终端/网络的情况下测试核心逻辑？
- [ ] 是否处理了异常输入和边界条件？
- [ ] 是否存在 secret 泄露到 stdout、stderr、log、临时文件的风险？
- [ ] 是否与现有命名、错误处理和目录风格保持一致？
- [ ] 新增功能是否避免了对旧模块的大面积侵入？

## 7. 合并门槛

任意一个功能分支要合并，至少应满足：

1. 核心功能有自动化测试覆盖。
2. 不引入新的明文 secret 暴露路径。
3. 模块边界比改动前更清晰，而不是更混乱。
4. 文档、命令帮助或配置示例与代码保持同步。
